🏗️ High-Level Design
Eight conceptual components and why each one must exist. Click any card to jump to its solution.
🔄
Ingestion Pipeline
Reads and validates all feedback JSON files from a mounted folder on batch trigger.
Why: The system needs a reliable entry point that handles malformed files gracefully without losing an entire batch.
MTP
🧠
AI Extraction Engine
Calls DeepSeek API once per feedback to identify school/service mentions and classify each as positive or negative.
Why: This is the core automation — it replaces the officer's manual reading and extraction entirely.
MTP
📊
CSV Generator
Produces two CSVs per school/service recipient: one for positive mentions, one for negative, each with the source feedback ID.
Why: Deans need a structured, separated view of feedback — not a dump — to draft their student responses.
MTP
✉️
Email Draft Generator
Generates a ready-to-send .eml file per approved recipient with CSVs attached. Officer sends manually.
Why: Removing SMTP from the system keeps the officer in the loop and legally responsible for what is sent.
MUP
🖥️
Web Application
FastAPI backend + Jinja2 + HTMX serving the officer review UI, batch management, audit log, and analytics dashboard.
Why: Officers and leadership need a browser interface — not a CLI — to review, approve, and monitor feedback processing.
MTPMUPMVP
🗄️
Data Store (PostgreSQL)
Stores raw feedbacks, extraction results, batch state, audit log, and users. Source of truth for all structured data.
Why: Relational structure is essential for batch state management, per-recipient approval tracking, and SQL analytics.
MTPMUPMVP
🔍
Vector Store (Qdrant)
Stores all-MiniLM-L6-v2 embeddings of feedback bodies for semantic reference search. Officers can query by meaning, not keyword.
Why: Keyword search is brittle for natural-language feedback. Semantic search lets officers ask "Wi-Fi complaints 2024" and find relevant records even if the word "Wi-Fi" isn't used.
MUPMVP
📈
Analytics Engine
SQL aggregation queries over the historical feedback store: sentiment trends per school/service over time, complaint volume rankings.
Why: Leadership cannot act on individual feedbacks — they need aggregated, time-series views to identify systemic issues and track improvement.
MVP
🐳
Containerisation
Docker Compose orchestrates three services: web (FastAPI), db (PostgreSQL), qdrant. Single docker compose up from local to on-prem.
Why: Reproducible environment across developer laptop and university on-prem server is non-negotiable for a demo-first product.

🔄
Ingestion Pipeline
Python · asyncio · pathlib · pydantic schema validation
MTP
›
A Python async function triggered by the officer's "Run Batch" action. It walks the mounted /feedback-data/<TYPE>/ directory, reads each .json file, validates it against a Pydantic schema (metadata + body structure), and persists valid records to PostgreSQL in bulk. Malformed files are logged with their filename and error reason — the batch continues without them. A batch record is created upfront so the UI can show processing status in real time via polling.
📋 Architecture Decision Record Accepted
Context
Officers have 50 feedback files per batch sitting on the local filesystem. The system needs to read them reliably, skip bad files, and persist them before making any API calls — so a crash mid-batch doesn't lose data.
Options Considered
✓ Chosen Docker volume mount + server-side file walk — officer selects folder path in UI, server reads directly from the mount.
✗ Rejected Browser file upload (multipart form) — uploading 50 × 1,200-word JSON files through the browser is slow and brittle; impractical for on-prem where files are already local.
✗ Rejected Watch folder with inotify daemon — adds always-on complexity; batch is triggered manually so no benefit.
Decision
Mount the host feedback directory into the container at /feedback-data via docker-compose.yml volume. Officer enters the subfolder path in the UI. Server walks it with pathlib.Path.glob("*.json"), validates with Pydantic, bulk-inserts to PostgreSQL. Same path works on local and on-prem — only the mount source changes.
Trade-offs
Pros
No upload overhead for large batches
Malformed files skipped without crashing batch
Same code path local → on-prem
Cons / Risks
Requires correct volume mount config per environment
Path must be configured in .env; typos cause silent empty batches
🧠
AI Extraction Engine
DeepSeek API · openai Python SDK (base_url override) · structured JSON output · asyncio gather
MTP
›
One DeepSeek API call per feedback using a system prompt that instructs the model to return a structured JSON array of mentions. Each mention contains: entity (school/service name), entity_type ("school" or "service"), extracted_text (verbatim section from the feedback), and sentiment ("positive" or "negative"). All 50 calls in a batch are dispatched concurrently with asyncio.gather(). Failed calls are retried once; persistent failures are logged and skipped.
📋 Architecture Decision Record Accepted
Context
Each ~1,200-word feedback mentions multiple schools and services across 10 Q&A pairs. We need reliable extraction of which entity is mentioned, what was said, and the sentiment — without 14+ API calls per feedback.
Options Considered
✓ Chosen Single prompt per feedback with structured JSON output schema — one DeepSeek call extracts all mentions at once.
✗ Rejected One call per entity (school/service) × 11 entities × 50 feedbacks = 550 API calls per batch. Latency and cost are unacceptable.
✗ Rejected Fine-tuned NER model — requires labelled training data we don't have; maintenance burden exceeds benefit for this scale.
Decision
Use DeepSeek via the openai Python SDK with base_url="https://api.deepseek.com". System prompt lists all known school/service names explicitly so the model doesn't hallucinate entity names. Response forced to JSON via response_format={"type":"json_object"}. Pydantic validates the response shape before DB insert.
Trade-offs
Pros
1 API call per feedback — 50 total per batch
Concurrent dispatch keeps batch time low (~10–30s for 50 files)
Swappable to any OpenAI-compatible LLM by changing base_url
Cons / Risks
Prompt engineering required; prompt must stay updated when new schools/services are added
Occasional JSON parsing failure requires retry logic
External API dependency — batch fails if DeepSeek is unreachable
📊
CSV Generator
Python csv module · in-memory generation · FastAPI StreamingResponse
MTP
›
Queries PostgreSQL for all extractions belonging to an approved batch recipient, groups by sentiment, and writes two CSVs: positive.csv (columns: feedback_id, extracted_text) and negative.csv (same columns). Generated in-memory on demand — no files written to disk. Served via StreamingResponse to the Email Draft Generator or directly to the officer as a download.
📋 Architecture Decision Record Accepted
Context
Deans need positive and negative mentions separated, each with the source feedback ID so they can reference-check the original file. The output must be re-generatable if the officer re-downloads.
Options Considered
✓ Chosen In-memory generation via Python csv stdlib, regenerated on each download request from DB data.
✗ Rejected Pre-generated files stored on disk — adds file lifecycle management (cleanup, uniqueness); unnecessary when DB is the source of truth.
✗ Rejected Excel (.xlsx) via openpyxl — heavier dependency; CSV is simpler and universally compatible with dean's email clients and tools.
Decision
Generate CSVs on demand from PostgreSQL extractions. Always reflects current approval state. Officer can re-generate after editing approvals without reprocessing.
Trade-offs
Pros
No disk storage management needed
Always current — reflects latest approval state
Zero extra dependencies
Cons / Risks
DB query on each download (negligible for 50-file batches)
✉️
Email Draft Generator
Python email.mime stdlib · RFC 2822 .eml · MIMEMultipart with CSV attachments
MUP
›
For each approved recipient, generates an .eml file containing: a pre-filled To header (from SCHOOLS_DEPARTMENTS.json), a structured HTML email body summarising the batch and mention counts, and the two CSVs as MIME attachments. The officer downloads the .eml, opens it in any email client (Outlook, Thunderbird, Gmail via drag-and-drop), and sends it — taking full responsibility for the content.
📋 Architecture Decision Record Accepted
Context
The system must not send emails directly. The officer is the responsible sender. But the system should eliminate all manual composition work — the officer should only need to open and click Send.
Options Considered
✓ Chosen Generate .eml file (RFC 2822) with CSVs as MIME attachments — officer opens in any email client and sends.
✗ Rejected SMTP integration — explicitly out of scope; removes officer control and legal accountability.
✗ Rejected Display-only HTML body for copy-paste — cannot carry file attachments; officer would need to attach CSVs manually.
Decision
Use Python's email.mime.multipart.MIMEMultipart to build the email, attach both CSVs as text/csv parts, pre-fill To, Subject, and Body. Serve as application/octet-stream download. Zero external dependencies.
Trade-offs
Pros
No SMTP server required
Works with any email client
Officer retains full send control
Zero additional dependencies
Cons / Risks
Officer must manually open .eml (by design — this is the review gate)

Web Application
FastAPI · Jinja2 templates · HTMX · SQLAlchemy ORM · Alembic migrations
MTPMUPMVP
›
Single FastAPI application serving all UI and API routes. Jinja2 renders server-side HTML — no separate frontend build. HTMX provides just enough interactivity: progress polling during batch extraction, inline approve/skip toggles, live search results. SQLAlchemy handles all DB access with Alembic managing schema migrations. At MTP it serves only the review UI; MUP adds approval and audit views; MVP adds analytics dashboard and role-gated access.
📋 Architecture Decision Record Accepted
Context
Need a Python web UI that handles async LLM API calls, renders a review dashboard, and ships as a single Docker container. The demo audience is officers and leadership — not developers — so the UI must work out of the box.
Options Considered
✓ Chosen FastAPI + Jinja2 + HTMX — async backend, server-rendered HTML, no separate JS build step.
✗ Rejected Flask — no native async support; LLM batch calls would block worker threads and freeze the UI during extraction.
✗ Rejected React/Next.js SPA + FastAPI API — requires a separate frontend container and build pipeline; overkill for this use case; adds Docker Compose complexity for a demo.
Decision
FastAPI for async LLM handling; Jinja2 for HTML templating with no build step; HTMX for interactivity via hx-get / hx-post without writing JavaScript. Single web container in Docker Compose.
Trade-offs
Pros
Async LLM calls don't block UI
Single Docker container, no build pipeline
HTMX interactivity without JavaScript framework
Cons / Risks
HTMX limits complex client-side state (acceptable here)
Server-side rendering is less snappy than SPA for analytics charts (use Chart.js CDN to compensate)

Data Store (PostgreSQL)
PostgreSQL 16 · SQLAlchemy ORM · Alembic · Docker volume for persistence
MTPMUPMVP
›
PostgreSQL stores five core tables: feedbacks (raw JSON body + metadata), batches (processing state + trigger info), extractions (per-mention extraction results), batch_recipients (approval state per school/service per batch), audit_log (immutable record of drafts generated). MVP adds users and roles. Indexes on (school_id, sentiment, batch_date) support analytics aggregation queries efficiently.
📋 Architecture Decision Record Accepted
Context
Need relational storage for batch state (with per-recipient approval tracking), structured extraction results queryable for analytics, and an audit log — plus multi-user access in MVP.
Options Considered
✓ Chosen PostgreSQL 16 in Docker Compose with named volume for persistence across container restarts.
✗ Rejected SQLite — write-locks under concurrent access break multi-user MVP; not suitable for on-prem deployment with multiple simultaneous users.
✗ Rejected MongoDB — document store adds no benefit over PostgreSQL's JSON column support; SQL aggregation for analytics is superior to MongoDB's aggregation pipeline for this use case.
Decision
PostgreSQL 16 official Docker image. Data persisted via named volume postgres_data. Alembic handles schema migrations so upgrades from MTP → MUP → MVP don't require manual SQL. SQLAlchemy ORM used throughout.
Trade-offs
Pros
ACID transactions for batch state
SQL analytics without extra infrastructure
Alembic migrations for MTP→MVP schema evolution
Cons / Risks
Heavier than SQLite (~200MB RAM) — acceptable given machine specs

Vector Store (Qdrant)
Qdrant Docker · qdrant-client Python SDK · sentence-transformers all-MiniLM-L6-v2 · 384-dim cosine similarity
MUPMVP
›
Qdrant runs as a separate service in Docker Compose. During ingestion, all-MiniLM-L6-v2 (loaded once at FastAPI startup via sentence-transformers) generates a 384-dimensional embedding for each feedback's body text. The vector is upserted to Qdrant alongside a payload: {feedback_id, type, school_id, related_services, sentiment, date}. At search time, the query is embedded with the same model and a filtered nearest-neighbour search is run. PostgreSQL provides full feedback content; Qdrant only holds vectors + payload for lookup.
📋 Architecture Decision Record Accepted
Context
Officers need to find relevant historical feedbacks by meaning (e.g. "students unhappy with registration process"). Keyword search fails when different words express the same complaint. Two decisions required: vector store choice and embedding model choice.
Options Considered
✓ Chosen Qdrant (separate Docker service) + all-MiniLM-L6-v2 (local, CPU) — production-ready, airgap-safe, zero API cost.
✗ Rejected pgvector extension on PostgreSQL — complicates PostgreSQL setup; mixes concerns; Qdrant's filtering and indexing is purpose-built and superior.
✗ Rejected DeepSeek embeddings API — requires outbound internet from on-prem server, which may be restricted at the university. Deployment blocker.
✗ Rejected Chroma — weaker persistence guarantees; less production-ready for on-prem than Qdrant.
Decision
Qdrant qdrant/qdrant Docker image with named volume qdrant_data. all-MiniLM-L6-v2 loaded at startup (~80MB). Embedding runs on CPU at ~50–200ms per feedback — 50 feedbacks ≈ 10–15s total, acceptable within a batch. RBAC filter applied at Qdrant query time via payload filter on school_id.
Trade-offs
Pros
Airgap-safe — no internet required for embeddings
Zero API cost for search
Qdrant persists across restarts with named volume
Filterable by school/type/sentiment at vector search time
Cons / Risks
384 dimensions vs 3072 (DeepSeek) — lower precision for very nuanced queries (acceptable for <10k feedbacks/year corpus)
~300MB additional RAM for model + Qdrant service
📈
Analytics Engine
SQLAlchemy SQL aggregation · Chart.js (CDN) · role-gated queries · CSV/PDF export
MVP
›
SQL aggregation queries over the extractions table — grouped by school_id or service_id, sentiment, and batch_date period — produce the two key views: sentiment trend over time and complaint volume ranking. Results are serialised to JSON and rendered client-side by Chart.js (loaded from CDN, no build step). Role-based filters are applied server-side: deans see only their school; leadership sees all. PDF export uses weasyprint; CSV uses Python csv stdlib.
📋 Architecture Decision Record Accepted
Context
Leadership needs sentiment trend charts and complaint rankings from historical data. The question is whether to add a dedicated analytics database or use PostgreSQL directly.
Options Considered
✓ Chosen Direct SQL aggregation queries on PostgreSQL with indexes — no additional infrastructure.
✗ Rejected Dedicated OLAP store (ClickHouse, DuckDB) — significant infrastructure overhead for <10,000 feedbacks per year; premature optimisation.
✗ Rejected Pandas in-memory aggregation — loads entire table into memory on each request; worse than SQL for filtered queries; no query optimisation.
Decision
SQL GROUP BY aggregations on indexed extractions table. Indexes on (school_id, sentiment, batch_date) added in MTP so data is index-ready at MVP. Chart.js renders trend line and bar charts from JSON endpoint. Materialized views added if query time exceeds 500ms as corpus grows.
Trade-offs
Pros
No additional Docker services
SQL is well-understood; easy to extend
Chart.js from CDN requires no build step
Cons / Risks
Query performance degrades with large datasets — mitigated by indexes and future materialized views

Containerisation
Docker Compose · 3 services: web, db, qdrant · .env config · named volumes
MTPMUPMVP
›
Three Docker Compose services: web (FastAPI app, built from Dockerfile), db (PostgreSQL 16 official image), qdrant (qdrant/qdrant official image). Named volumes postgres_data and qdrant_data persist data across restarts. .env file controls DEEPSEEK_API_KEY, FEEDBACK_DATA_PATH, DATABASE_URL. Same docker-compose.yml works on local dev machine and on-prem university server — only the volume source path and .env values change.
📋 Architecture Decision Record Accepted
Context
Product must demo on a local developer machine and then deploy identically to an on-prem university server with no DevOps infrastructure. Environment reproducibility is non-negotiable.
Options Considered
✓ Chosen Docker Compose — single docker compose up starts the full stack on any machine with Docker installed.
✗ Rejected Kubernetes — massive operational overhead for a 3-service application at <30 concurrent users; university IT team is unlikely to have Kubernetes expertise.
✗ Rejected Bare-metal / virtualenv deployment — environment drift between developer machine and on-prem server causes "works on my machine" failures; no isolation between system Python and app dependencies.
Decision
docker-compose.yml with three services. MTP ships with web + db. MUP adds qdrant. All environment variables in .env (gitignored). Volume mounts for feedback data configurable per environment. sentence-transformers model downloaded into image at build time so it is available offline on on-prem server.
Trade-offs
Pros
Single command to start full stack
Identical environment local → on-prem
Model baked into image — works offline after first build
Cons / Risks
Docker image with sentence-transformers model is ~1.5GB — one-time download at build
No horizontal scaling (not needed at this user count)
