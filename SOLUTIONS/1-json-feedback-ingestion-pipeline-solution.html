Problem Statement

What specific problem are we solving?
Officers must manually open and process each feedback JSON file — one by one — to extract and forward content. This is the primary automation target: the system needs a single triggered action that reads an entire batch folder, validates every file, and persists all valid records to the database. A single malformed file must not abort the batch; errors must be logged and skipped transparently.

Implementation Drifts

What deviated from the plan?
Drift 1 — JSON vs JSONB
Solution doc specified JSONB (PostgreSQL-only). Changed to JSON type so tests run on SQLite in-memory without Docker. Acceptable at MTP; switch to JSONB with GIN index at MVP if JSON query performance becomes a concern.
Drift 2 — No Alembic at MTP
Solution doc mentioned Alembic migrations. Using Base.metadata.create_all() at startup instead. Simpler for MTP where there is no prod data to migrate. Alembic added before MVP when schema evolution matters.
Drift 3 — _run() extracted for testability
walk_and_ingest() creates its own DB session (required for BackgroundTask isolation). Tests call the internal _run() directly with a fixture session to avoid spawning real DB sessions. Keeps tests fast and session-safe.
Drift 4 — Persona shape varies by feedback type
A single shared persona schema assumed every persona has a plain `name` field. Real DUMMY_DATA proved otherwise: PARENTS adds `child_name`, INTERNSHIP has no `name` at all (uses `evaluator_name`/`intern_name` instead). This silently failed validation for 100% of INTERNSHIP files (50/50) — the entire feedback type was being dropped. Fixed by dispatching to a per-type persona schema (StudentPersona/ParentPersona/EmployerPersona) keyed off `metadata.type` via a model_validator. Verified against all 200 real DUMMY_DATA files (0 failures) and confirmed the bug was real by reverting the fix and re-running (50/50 INTERNSHIP files failed on old code).
